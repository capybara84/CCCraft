# KNOWLEDGE - ドメイン知識・調査結果

## 技術的な知見
- cannon-esでプレイヤー速度を直接制御する場合、摩擦を0にしないと水平移動が極端に遅くなる
- Canvas APIによるプロシージャルテクスチャはNearestFilterで描画するとピクセルアート風になる
- DDAレイマーチングのmaxDistはカメラ距離+操作範囲が必要（3rdパーソンではカメラからレイが飛ぶため）
- チャンクメッシュ単位の遮蔽半透明は粗すぎる → シェーダーのonBeforeCompileでピクセル単位フェードが最適
- Three.jsのBoxGeometryの面順は +X, -X, +Y, -Y, +Z, -Z（マルチマテリアル配列のインデックスに対応）
- テクスチャアトラスのUV: canvasはY下向き、Three.jsはY上向き → v = 1 - (row+1)/atlasSize
- 入力の左右分割はiPad互換に見えて実は不便（右半分のオブジェクトを操作できない）→ クリックvsドラッグ判別が正解

## Phase 4で得た知見

### cannon-es重力相殺
- 飛行中に重力を無効にするには `body.force.y = body.mass * GRAVITY` で反力を毎フレーム加える
- body.gravityの直接変更やbody.type変更より安全（他のオブジェクトに影響しない）
- 飛行解除時に `body.force.y = 0` で即座に重力復帰

### 飛行起動のジャンプ
- toggle()で速度を設定してもupdate()が同フレームで上書きする問題がある
- `_needsLaunch`フラグで初フレームの速度設定を確実にするパターンが有効
- `_rising`状態で自然な放物線ジャンプ→最高点到達後に浮遊へ移行

### Three.js腕アニメーション
- BoxGeometryのrotation.z: 負=左方向に広がる、正=右方向に広がる
- 左腕はrotation.z負、右腕はrotation.z正で横に広げる
- rotation.z = 0.8程度が自然な広がり（1.2だと離れすぎ、0.3だと控えめすぎ）
- 飛行→通常遷移時に `rotation.z *= 0.8` のdecayで滑らかにリセット

### 移動方向の計算
- W=z:-1, A=x:-1 の入力をカメラyawで回転する式:
  - worldDirX = input.x * cos(yaw) + input.z * sin(yaw)
  - worldDirZ = -input.x * sin(yaw) + input.z * cos(yaw)
- 飛行モードも通常モードも同じ式を使わないと方向がズレる

## 決定事項と理由
- カメラ俯角35度、距離10（より水平に近い視点でキャラの前髪や顔が見える）
- 通常=走り、Shift=歩き（普段のプレイで走る方が多いため）
- 接地ジャンプのみ（飛行モードで自由飛行できるため空中ジャンプは不要）
- 操作範囲5ブロック（10は遠すぎる、3rdパーソンでも5で十分）
- テクスチャシステム: 頂点カラー → アトラス+UVに移行（てんてん模様でディテイル向上）
- 遮蔽フェード: uFadeRadius=0.8基準、カメラ側×2.0/プレイヤー側×1.0で線形補間、smoothstepでフェード、0.05未満はdiscard
- 破壊エフェクト: 進捗に応じて1-3個/フレームの粒子 + 完了時30個バースト
- 破壊進捗オーバーレイ: 8段階のクラックテクスチャをCanvasで生成、polygonOffsetでZファイティング防止
- InventoryUI: バックパックスロット選択→別スロットクリックでスワップ方式
- 飛行モード: グライダー（下降+水平）→ 自由飛行（Gトグル、カメラ方向3D移動、スタミナ/上昇気流なし）に変更
- カメラピッチ範囲: -60〜80度（飛行で急降下・急上昇が必要）
